# Stash Devlog

## 2026-05-15 — Plan/execute mode awareness added to agent system prompt

### What was built

**Plan-mode and execute-mode system prompt blocks**

Two new string constants were added to `stash/prompts/prompt.py`:

- `PLAN_MODE_BLOCK` (lines 74–90): appended to the system prompt when the agent is running in dry-run mode. It explains to the model that write tools will not execute, that `"[plan mode — not executed]"` responses are expected and should not be retried, that read tools run normally, and that the agent should conclude planning with a summary of what will happen upon approval.
- `EXECUTE_FROM_PLAN_BLOCK` (lines 92–102): appended to the system prompt when the agent resumes from a cached plan into live execution. It recontextualises the `"[plan mode — not executed]"` strings present in the message history — which the model would otherwise interpret as tool errors — and instructs the model to re-execute those steps now that write tools are live.

`build_system_prompt()` (line 105) gained a `mode` parameter (`"default"` / `"plan"` / `"execute"`, defaulting to `"default"`). The function appends the appropriate block before the optional user-preferences section. Signature change is backwards-compatible; all existing callers that omit `mode` continue to receive the unmodified prompt.

**Mode routing in `Agent._loop()`**

`Agent._loop()` in `stash/core/agent.py` (lines 76–86) was updated in two places:

1. When building the initial message list from scratch (no `initial_messages` provided), `mode` is now derived from `dry_run`: `"plan"` when `dry_run=True`, `"default"` otherwise. This is passed directly to `build_system_prompt()` at line 84.

2. When resuming from a cached plan (`initial_messages` provided), the system message at `messages[0]` is replaced with a freshly built `mode="execute"` prompt (line 79). The rest of the plan-phase message history — all the intermediate tool calls and their `"[plan mode — not executed]"` responses — is preserved verbatim. Only the system message is swapped.

No changes were made to `Agent.plan()`, `Agent.run()`, `app.py`, or any tool module.

---

### Decisions taken

**System message swap at `messages[0]`, not a new injected user message**

The alternative considered was appending a `{"role": "user", "content": "Plan approved. Please execute the steps now."}` message at the start of the execution phase. This was rejected because it changes the conversational structure: the model may respond to the injected user turn with another text summary rather than immediately calling tools. Swapping `messages[0]` is transparent to the conversation flow — the model sees the same history it produced during planning, with only the system-level framing changed.

**Mode expressed as a string literal, not a boolean**

`build_system_prompt(plan_mode: bool)` was considered. A string enum (`"default"` / `"plan"` / `"execute"`) was chosen instead because `plan` and `execute` are not inverses of each other — a caller setting `plan_mode=False` would be ambiguous between "regular chat" and "executing from a plan." Three named modes make the intent unambiguous at each call site.

**Plan-mode context kept in the system prompt, not in per-observation strings**

An earlier candidate fix was to embed mode instructions directly in each skipped-tool observation: e.g. `"[plan mode — not executed. Do not retry this step during planning.]"`. This was rejected for two reasons. First, it only addresses the retry-loop problem at the individual tool-call level; it does not give the model a coherent global understanding of the plan/execute lifecycle, which matters for multi-step tasks where the model needs to reason about dependent operations (rename → move → verify) as a sequence. Second, it does not solve the execution-phase recontextualisation problem at all — the model in execution still has no instruction about what the historical `"[plan mode — not executed]"` strings mean.

**System prompt swap done inside `_loop()`, not in `app.py`**

The handoff from plan approval to execution happens in `app.py`'s `on_plan_approved()` (line 250), which passes `pending.messages` to `agent.run()`. The swap could have been done there before the call. It was placed inside `_loop()` instead, at the point where `initial_messages` is detected, because that is the only place that has full context about both `dry_run` state and the `_load_preferences()` method needed to reconstruct the system prompt. Doing it in the app layer would require either exposing `_load_preferences()` publicly or duplicating the preference-loading logic.

### Problems identified

**Plan-phase context inconsistency: the original edge case**

During a plan (dry-run) run, write tools return `"[plan mode — not executed]"` while read tools execute against the real filesystem. This creates a state mismatch: if the agent calls `rename("file1.txt", "file2.txt")` and then `ls("/dir")`, it sees `"[plan mode — not executed]"` from the rename and `file1.txt` still present from the ls. Without global plan-mode awareness in the system prompt, a weaker model may interpret the rename response as a failure and retry it, potentially looping until `max_steps` is reached (`agent.py` line 150–153).

**Status:** Resolved. `PLAN_MODE_BLOCK` explicitly instructs the model not to retry write tools that return `"[plan mode — not executed]"`, and to treat this as confirmation that the intent has been recorded.

**Plan-mode context bleed-through into execution**

The plan caching mechanism (`agent.py` lines 76–80) reuses the full `messages[]` array from the plan phase as `initial_messages` for execution. Any mode instruction embedded in `messages[0]` during planning would therefore persist into execution. A naïve fix of injecting `"You are in PLAN MODE"` into the system prompt during planning would cause the model to believe it is still planning during the execution phase.

**Status:** Resolved. `messages[0]` is unconditionally replaced when `initial_messages` is present and `_loop()` is running with `dry_run=False`. The plan-mode system message never survives into execution.

**Execution phase: historical `"[plan mode — not executed]"` strings misread as errors**

Without intervention, the execution-phase model sees a conversation history full of `"[plan mode — not executed]"` tool responses. The existing system prompt (`SYSTEM_PROMPT`, line 37) instructs the model: "If a tool returns an error — explain what went wrong and ask how the user wants to proceed." The model could reasonably classify these historical strings as errors and ask the user for guidance rather than re-executing the planned steps.

**Status:** Resolved. `EXECUTE_FROM_PLAN_BLOCK` explicitly tells the model these strings are from the planning phase, not live errors, and instructs it to re-execute the corresponding steps.

### Open questions

- **Does `EXECUTE_FROM_PLAN_BLOCK` produce correct behaviour with weaker Ollama models (e.g. `qwen2.5:1.5b`)?** The instruction to re-execute steps from history is relatively high-level. A model with limited instruction-following capability may still summarise the plan as text rather than calling tools. This has not been tested against actual model runs.

- **Should the `mode` parameter on `build_system_prompt()` be typed as a `Literal["default", "plan", "execute"]`?** Currently it is typed as `str`, which means invalid values are silently ignored (the function falls through to the default behaviour). A `Literal` type would catch invalid callers at static analysis time.

- **Is there a scenario where `initial_messages` is passed to `_loop()` with `dry_run=True`?** The current code unconditionally swaps `messages[0]` to `mode="execute"` whenever `initial_messages` is present, without checking `dry_run`. No such call path exists today — `Agent.plan()` never passes `initial_messages` and `Agent.run()` always uses `dry_run=False` — but the assumption is implicit rather than enforced.

- **Should the per-observation string `"[plan mode — not executed]"` be updated to match the language in `PLAN_MODE_BLOCK`?** The block references this exact string. If one is changed without the other, the model's in-prompt instruction would reference a string it will never see, weakening the guidance. These two strings are currently maintained separately with no shared constant.
