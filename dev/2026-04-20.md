## 2026-04-20 — Live testing: startup crash fixes, screen stack overhaul, conversation correctness, and UI polish

### What was built

This session was a live testing pass — no new features were added. Every change was a bug fix uncovered by running the application for the first time end-to-end. The fixes span the TUI startup lifecycle, screen stack architecture, conversation history correctness, config key consistency, and chat UI presentation.

---

**`stash/tui/screens/loading.py` — timer callback return value**

`LoadingScreen.on_mount()` (line 259) used a lambda to schedule its own dismissal:

```python
self.set_timer(5.0, lambda: self.dismiss(self._health_result))
```

`Screen.dismiss()` returns an `AwaitComplete` object. The lambda implicitly returns this value. Textual's internal `invoke()` function in `_callback.py` checks `isawaitable(result)` on every callback return value and, if truthy, does `await result`. This triggered `dismiss()`'s internal `pre_await` guard, which raises `ScreenError` if the active message pump is the screen being dismissed. The fix replaces the lambda with a named function that explicitly returns `None`, preventing `invoke()` from awaiting the `AwaitComplete`:

```python
def on_mount(self) -> None:
    def _dismiss() -> None:
        self.dismiss(self._health_result)
    self.set_timer(5.0, _dismiss)
```

**`stash/tui/app.py` — screen stack architecture**

`StashApp.compose()` previously yielded `MainScreen()`. Textual's screen stack starts empty (`_screen_stacks: dict[str, list[Screen]] = {DEFAULT_MODE: []}` in `app.py` of the Textual package). `App.compose()` mounts widgets into the app layer, not the screen stack. `MainScreen` was therefore never a reachable screen — it existed in the DOM but was invisible behind modal overlays.

The consequence: after `LoadingScreen` dismissed (popping to an empty stack) and `ModelPickerScreen` subsequently dismissed, the stack was empty and the display went dark.

`compose()` was removed entirely. `on_mount()` now pushes `MainScreen` explicitly before pushing `LoadingScreen`, establishing a persistent base screen:

```python
def on_mount(self) -> None:
    from stash.tui.screens.loading import LoadingScreen
    model = self.config.get("ollama", {}).get("model", "")
    self.push_screen(MainScreen(model=model))
    self.push_screen(LoadingScreen(self._health_result), self._on_loading_done)
```

Stack after `on_mount`: `[MainScreen, LoadingScreen]`. After loading: `[MainScreen]`. After model picker: `[MainScreen]`. `MainScreen` is never popped.

The model string from config is passed at construction so the `TitleBar` shows the correct model immediately on subsequent launches without waiting for a callback.

**`stash/tui/app.py` — `on_ollama_status_changed` screen query**

`on_ollama_status_changed()` (line 286) called `self.screen.query_one("TitleBar")`. `self.screen` returns the topmost active screen. When `ModelPickerScreen` is on the stack, `self.screen` is `ModelPickerScreen`, which has no `TitleBar` — raising `NoMatches`. The fix iterates `self.screen_stack` to locate `MainScreen` explicitly:

```python
for screen in self.screen_stack:
    if isinstance(screen, MainScreen):
        screen.query_one("TitleBar").set_ollama_status(message.available)
        break
```

The same pattern was applied to `_apply_model_to_ui()` (line 327) for the same reason.

**`stash/tui/app.py` — `_on_model_selected` callback timing**

`Screen.dismiss()` fires result callbacks synchronously before calling `pop_screen()`. When `_on_model_selected` was called, `self.screen` was still `ModelPickerScreen`. The original `self.screen.query_one("TitleBar").set_model(model)` therefore raised `NoMatches`, and because the exception propagated before `_save_config()`, the chosen model was never persisted to disk.

The fix separates the two concerns: config is saved immediately (synchronous, safe at any point), and the UI update is deferred via `call_later()`, which runs after all pending messages — including `pop_screen()` — have been processed:

```python
def _on_model_selected(self, model: str | None) -> None:
    if not model:
        return
    self.config.setdefault("ollama", {})["model"] = model
    self._save_config()
    self.call_later(self._apply_model_to_ui, model)
```

`call_after_refresh` was considered but rejected: it fires after the next DOM refresh, which may precede the screen pop completing. `call_later` runs after the message queue drains, which is the correct ordering guarantee here.

**`stash/tui/app.py` — no-action-steps response path**

When the agent returns no `action`-type steps — a conversational reply, a clarifying question, or a guardrail refusal — the original code unconditionally called `show_plan()`, mounting a `PlanMessage` with an empty tool list and an empty step table. The user was asked to approve a plan that contained nothing.

A branch was added in `on_task_submitted()` (line 213) to detect this case:

```python
has_actions = any(s.type == "action" for s in steps)
if not has_actions:
    chat.remove_planning_bubble()
    for step in steps:
        chat.append_step(step)
    ...
    self._run_state = RunState.IDLE
    return
```

Response steps are rendered directly into the stream and the state machine returns to `IDLE` without entering `AWAITING_APPROVAL`.

**`stash/core/agent.py` — conversation history in plan mode**

`Agent._loop()` previously loaded session history only when `dry_run=False`:

```python
if not dry_run:
    messages.extend(self._load_session_history())
```

Plan mode (`dry_run=True`) started every call with only the system prompt and the current user message, making it stateless. This broke clarification flows: a user's follow-up answer to a clarifying question arrived in a fresh context with no record of the prior exchange. The guard was removed; history is now loaded unconditionally (line 76).

---

**`stash/tui/app.py` — user message persistence timing**

`db.add_message("user", task)` was called before `agent.plan()` ran. This caused the current user message to appear in `_load_session_history()` during the subsequent `agent.run()` call (which loads history), and then to be appended again explicitly — resulting in the same user message appearing twice at the end of the run-phase context. The call was moved to after `agent.plan()` completes successfully (line 211), ensuring history reflects only completed exchanges.

---

**`stash/tui/app.py` — assistant response persistence**

For the no-action-steps path, the agent's response was displayed in the UI but never written to the `conversations` table. The next turn would therefore have no record of what the agent said. A `db.add_message("assistant", ...)` call was added to the no-action branch (line 221), mirroring what `on_plan_approved()` already did for the full execution path (line 249).

---

**`stash/core/agent.py` — config key mismatch**

`Agent.__init__()` read the Ollama host from `ollama_config.get("endpoint", ...)` (line 51). The config written by `main.py`'s `_DEFAULT_CONFIG_TOML` uses the key `host`, as does the health check in `main.py` (line 164: `config.get("ollama", {}).get("host", ...)`). The agent was therefore always falling back to the hardcoded default `"http://localhost:11434"` on fresh installs, ignoring any user-configured value. Fixed to `ollama_config.get("host", "http://localhost:11434")`.

---

**`stash/persistence/sqlite.py` — SQLite thread safety**

`db.connect()` opened the connection without `check_same_thread=False`. Once history loading was added to plan mode, `_load_session_history()` was called from a `ThreadPoolExecutor` thread (via `run_in_executor`), while the connection was created in the Textual event loop thread. Python's `sqlite3` raises `ProgrammingError` on cross-thread use by default. The database was already using WAL mode (`PRAGMA journal_mode=WAL`), which permits concurrent reads and writes from separate connections; enabling `check_same_thread=False` on the shared connection (line 23) is safe in this context.

---

**`stash/tui/app.py` — plan failure leaves UI in broken state**

The `except` block in `on_task_submitted()` handled plan failures by logging, marking the run failed, and setting `_run_state = IDLE`. It did not remove the "planning…" bubble or re-enable the input. The user was left with a disabled input and a stale system bubble with no indication of what had gone wrong. The error path now calls `chat.remove_planning_bubble()`, mounts an error bubble, and calls `chat.set_input_enabled(True)` before returning (lines 206–208).

---

**`stash/tui/widgets/chat.py` — "planning…" bubble persists**

`on_task_submitted()` mounted a system bubble with the text "planning…" at the start of every task. Nothing removed it. After the response arrived, the bubble remained as a visual artefact between every user and agent turn.

`MessageBubble.__init__()` (line 135) was updated to accept `**kwargs` and forward them to `Widget.__init__()`, allowing a widget `id` to be passed. `ChatWidget.append_bubble()` (line 410) was extended with an optional `bubble_id` parameter. A `remove_planning_bubble()` method (line 416) was added, which calls `.remove()` on `#planning-bubble` inside a try/except. The bubble is now tagged on creation and removed in three places: the no-action path, the plan-failure error path, and `show_plan()` (line 447), so the plan card replaces rather than follows the "planning…" indicator.

---

**`stash/tui/widgets/chat.py` — chat bubble colour differentiation**

User (`MessageBubble.user`) and agent response (`MessageBubble.response`) bubbles shared identical styling: `#58A6FF` blue for both the left border accent and the header label. The only distinction was the header text ("you" vs "stash — response"), which was insufficient at a glance.

Agent response bubbles were changed to `#FF7B72` — the GitHub Dark keyword colour, which sits on the opposite side of the colour wheel from the existing blues and greens. The change applies to both the `border-left` rule (line 108) and the `#msg-header` colour rule (line 117). No other bubble types were affected.

### Decisions taken

**Remove `compose()` from `StashApp`, push `MainScreen` in `on_mount()`**

The alternative was to use Textual's `SCREENS` class variable and an `INITIAL_SCREEN` declaration. This was not chosen because the startup sequence has a runtime dependency — the model string from config must be passed to `MainScreen` at push time — which does not fit the static `SCREENS` dict pattern cleanly. Pushing explicitly in `on_mount()` keeps the full startup logic in one readable sequence.

**`call_later` over `call_after_refresh` for TitleBar update after model selection**

`call_after_refresh` schedules for the next paint cycle, which may fire before `pop_screen()` completes its internal state transitions. `call_later` runs after the message queue drains, providing the ordering guarantee needed to ensure `ModelPickerScreen` is no longer the active screen when `_apply_model_to_ui` queries the stack.

**History loaded unconditionally in `Agent._loop()`, user message saved after plan**

The original two-phase design (plan stateless, run stateful) was likely intended to keep plan fast and avoid influencing the dry-run with prior conversation. However, the system prompt explicitly describes conversational exchanges as part of the agent's behaviour, making stateless planning incoherent for multi-turn clarification flows. History is now loaded in both phases. The user message is saved after a successful plan rather than before, to prevent it appearing twice in run-phase context.

**`check_same_thread=False` on the shared SQLite connection**

The alternative was to open a separate connection per executor thread. This would eliminate shared-state concerns entirely but would require connection lifecycle management across the executor pool and potential write conflicts (mitigated but not eliminated by WAL). Since the cross-thread access pattern is read-only from the executor (history loads) and write-only from the main thread (audit logs, message saves), and WAL mode is already active, the shared connection with `check_same_thread=False` is the simpler and appropriate choice here.

**`#FF7B72` for agent response bubbles**

The alternatives considered were teal (`#39C5CF`) and warm gold (`#E8B85D`). Teal was rejected as too close to blue at terminal colour depths. Gold was rejected on aesthetic grounds by the project owner. `#FF7B72` was chosen for its semantic legibility (it is the GitHub Dark keyword colour, familiar to developers) and its position on the opposite side of the colour wheel from `#58A6FF`, making the user/agent distinction immediately readable.

### Problems identified

**`ScreenError` on startup — resolved**

`LoadingScreen.on_mount()` used a lambda that returned `Screen.dismiss()`'s `AwaitComplete`. Textual's `invoke()` awaited this, triggering `dismiss()`'s `pre_await` guard. Fixed in `stash/tui/screens/loading.py:260`.

**Empty screen after model picker — resolved**

`App.compose()` does not place widgets on the screen stack. `MainScreen` was never in the stack, so after all modal screens dismissed the stack was empty. Fixed by pushing `MainScreen` first in `on_mount()`, `stash/tui/app.py:135`.

**`NoMatches` on `TitleBar` when `OllamaStatusChanged` fires during `ModelPickerScreen` — resolved**

`self.screen` returns the topmost screen. Fixed by iterating `self.screen_stack` in `on_ollama_status_changed()` and `_apply_model_to_ui()`, `stash/tui/app.py:288–291` and `329–332`.

**Model name not persisted on first selection — resolved**

`_on_model_selected()` raised before reaching `_save_config()` because `self.screen.query_one("TitleBar")` failed. Config save now precedes the UI update, `stash/tui/app.py:320–321`.

**Plan failure leaves UI permanently broken — resolved**

The except block in `on_task_submitted()` did not clean up UI state. Fixed at `stash/tui/app.py:206–208`.

**SQLite cross-thread `ProgrammingError` — resolved**

Adding history loading to plan mode introduced a cross-thread SQLite access. Fixed by adding `check_same_thread=False` to `db.connect()` in `stash/persistence/sqlite.py:23`.

**"planning…" bubble not removed after response — resolved**

Fixed via `remove_planning_bubble()` in `stash/tui/widgets/chat.py:416`, called in three locations.

**Conversational responses sent to empty approval UI — resolved**

No-action-steps path added in `on_task_submitted()`, `stash/tui/app.py:213–225`.

**`endpoint` vs `host` config key mismatch in agent — resolved**

`Agent.__init__()` read `endpoint`; config writes `host`. Fixed in `stash/core/agent.py:51`.

**User message double-appended in run-phase context — resolved**

`db.add_message("user", task)` moved to after `agent.plan()` completes, `stash/tui/app.py:211`.

**Assistant response not persisted for conversational turns — resolved**

`db.add_message("assistant", ...)` added to the no-action branch, `stash/tui/app.py:221–222`.

**`ApproveBar` keyboard hints not implemented — open**

The `ApproveBar` displays "enter · esc" as a key hint label (`stash/tui/widgets/chat.py:287`) but neither binding is defined. Approve and reject are button-only. No keyboard shortcut currently triggers `PlanApproved` or `PlanRejected`.

### Open questions

1. **Should the executor use a dedicated SQLite connection per thread?** The current approach shares one connection with `check_same_thread=False`. WAL mode makes this safe for the current read/write split, but if write operations are ever added to the executor path, contention becomes a real concern. Is a connection pool or per-thread connection the right long-term answer?

2. **Should `agent.plan()` load history?** The current decision loads history unconditionally for conversational continuity. However, plan mode is a dry run — it does not execute write tools. Does exposing prior conversation context to the planning phase introduce any undesirable influence on what the agent proposes to do? Should history loading in plan mode be opt-in per rule?

3. **Why does `ApproveBar` advertise "enter · esc" bindings that do not exist?** Were these intended and never implemented, or are they aspirational UI copy? Should `enter` approve and `esc` reject while the approval bar is visible?

4. **Is Qwen2.5 1.5B a reasonable default for testing?** The model took 128 seconds to produce its first response in this session and runs entirely on CPU. The ReAct loop with tool calling requires a model capable of reliably emitting structured tool call JSON. At 1.5B parameters, failure modes (hallucinated tool names, malformed arguments) are likely to surface quickly. Gemma 3 or a larger Qwen variant would be a more reliable baseline for functional testing.

5. **Should the tests be updated to reflect the new screen stack architecture?** The `_test_on_mount` fixture in `stash/tests/test_tui.py` does not push `MainScreen`, meaning tests run against a screen stack that diverges from production. Screen-level queries in tests (e.g. `pilot.app.screen.query_one(...)`) will silently target the wrong screen if the stack is empty.
