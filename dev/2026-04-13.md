## 2026-04-13 ‚Äî Full TUI layer built and wired; prototype debugging session

### What was built

**Shared message module (`tui/messages.py`)**
Three message types ‚Äî `TaskSubmitted` (line 11), `PlanApproved` (line 18), `PlanRejected` (line 22) ‚Äî were extracted into a standalone `stash/tui/messages.py`. They had previously lived in `tui/app.py`, but `ChatWidget` needs to post them and `StashApp` needs to handle them, creating a circular import. Moving them to a neutral module breaks the cycle: `ChatWidget` imports from `messages`, `StashApp` imports from `messages`, and neither imports the other.

**`TitleBar` widget (`tui/widgets/title_bar.py`)**
The top bar of the app. Three live badges are managed as `Label` widgets with `display: none` as default and shown conditionally:

- `#ollama-badge` ‚Äî colour-coded pill: green for online, red for offline, amber for unknown. Driven by `set_ollama_status(available: bool)` (line 105). A `set_interval(0.8, _tick_pulse)` timer at line 99 alternates the dot glyph between `‚óè` and `‚óâ` ‚Äî a low-cost animated pulse that requires no CSS animation support.
- `#model-badge` ‚Äî shown/hidden by `set_model(model: str)` (line 109). Starts hidden; becomes visible once a model is selected.
- `#rules-badge` ‚Äî shown/hidden by `set_rule_count(count: int)` (line 118). Displays `‚ó∑ N rule(s)` in amber.

`TitleBar.__init__` accepts `model: str = ""` (line 84) so `MainScreen` can forward the initial model string at construction time.

**`ChatWidget` and sub-widgets (`tui/widgets/chat.py`)**
The chat column is composed of five nested widget classes:

- `PaneHeader` (line 50) ‚Äî a two-cell header row showing an uppercase column title and a keybind chip. Shared pattern used in both chat and sidebar.
- `MessageBubble` (line 78) ‚Äî renders one streamed message with a coloured left border. Type is passed as a CSS class (`.user`, `.thought`, `.action`, `.observation`, `.final`, `.error`, `.system`) so all styling is expressed in CSS rather than conditional Python. `_HEADERS` dict (line 125) maps type ‚Üí label string.
- `PlanStepRow` (line 152) ‚Äî one row in the plan table: `{index}. [tool chip] {description} ‚óã`. `mark_done()` (line 179) updates the status label to `‚úì`. Rows are assigned IDs as `plan-step-{index}` for targeted querying.
- `PlanMessage` (line 183) ‚Äî the inline plan card shown after planning completes. `__init__` pre-filters to action steps only and de-duplicates tool names with `dict.fromkeys()` (line 214) to build the tool chip row. `mark_step_done(index)` (line 228) delegates to the child `PlanStepRow`.
- `ApproveBar` (line 239) ‚Äî hidden by default (`display: none`). `show_bar()` / `hide_bar()` toggle it. Posts `PlanApproved` or `PlanRejected` from `on_button_pressed` (line 295). Because `ApproveBar` is a child of `ChatWidget`, these messages bubble up through `ChatWidget` ‚Üí `MainScreen` ‚Üí `StashApp` where the handlers live.
- `InputArea` (line 307) ‚Äî `‚Ä∫` prefix label + `Input`. `on_input_submitted` (line 340) strips the value, clears the field, and posts `TaskSubmitted`. `set_enabled(enabled)` (line 346) sets `inp.disabled = not enabled` and calls `inp.focus()` when enabling.
- `ChatWidget` (line 357) ‚Äî orchestrates all of the above. The public API used by `StashApp` is documented at line 366. `on_mount` calls `set_enabled(True)` (line 397) ‚Äî though this call may be discarded if `LoadingScreen` is already on top, which is resolved by the `_restore_main_focus` mechanism in `StashApp`.

`append_step(step: ReActStep)` at line 408 handles the five step types. For `action` steps it also calls `mark_step_done(self._action_step_idx)` and increments the counter, so plan step circles tick off in real time as the agent executes.

**`SidebarWidget` and sub-widgets (`tui/widgets/sidebar.py`)**
The right sidebar is split into two sections:

- `RuleItem` (line 92) ‚Äî displays one `FolderRule`. Status dot in column 1, name in column 2, `every {N}h ¬∑ {tools}` meta line below. `set_status(status)` (line 124) updates just the dot label. `_rule_status()` (line 38) derives display status from `rule.enabled` and `rule.last_run_status`.
- `FolderRulesSection` (line 132) ‚Äî holds the rules list. `load_rules()` (line 144) removes all existing `RuleItem` children and remounts from the new list ‚Äî no diffing, full replace.
- `AuditLogSection` (line 161) ‚Äî a `RichLog` widget (line 177) with `markup=True` and `auto_scroll=True`. `append_entry(tool, result)` (line 179) prepends a `HH:MM` timestamp then a green tool chip and result text.
- `SidebarWidget` (line 189) ‚Äî thin facade delegating all calls to the appropriate section. `focus_audit_log()` (line 215) focuses the `RichLog` directly so the user can scroll with keyboard.

**`LoadingScreen` (`tui/screens/loading.py`)**
A `Screen[HealthResult | None]` ‚Äî note the type parameter, which makes `dismiss(value)` return the `HealthResult` to the `_on_loading_done` callback in `StashApp`. Three animated sub-widgets:

- `_AsciiArt` (line 72) ‚Äî renders the STASH block letters using six `Label` rows. A `set_interval(1/20, ...)` timer runs at 20 fps (line 95). Each tick advances `_tick`, computes a `cycle_pos` from `_tick % 80 / 80`, then assigns a per-row phase offset from `_OFFSETS` (line 55). `_sweep_color(phase)` (line 33) interpolates across three colour stops ‚Äî blue ‚Üí purple ‚Üí green ‚Üí blue ‚Äî via linear interpolation. The gradient flows continuously downward across rows.
- `_BootProgressBar` (line 105) ‚Äî a reactive `progress` field drives `watch_progress`, which sets `styles.width` as a percentage string. Uses `Static` as the fill element rather than a styled container to avoid border-box sizing issues.
- `_StatusSection` (line 131) ‚Äî four status messages timed to fire at `[0.2, 1.3, 2.5, 3.8]` seconds (line 64). Each message fades in with `styles.animate("opacity", 1.0, duration=0.25)` and fades out after `_HOLD = 0.7` seconds except the last. The progress bar is animated in parallel.

`on_mount` (line 259) sets a single five-second timer: `self.set_timer(5.0, lambda: self.dismiss(self._health_result))`. No separate pop ‚Äî `dismiss()` handles popping automatically and calls the registered callback.

**`ModelPickerScreen` (`tui/screens/model_picker.py`)**
A `ModalScreen[str | None]`. Two layouts chosen at compose-time based on whether `self._models` is populated:

- Models found: a `ListView` with one `ListItem` per model name. `on_mount` (line 166) sets `lv.index` to the position of the currently configured model so the cursor lands on it. `action_select` (line 178) reads `lv.index` and dismisses with the model name. `on_list_view_selected` (line 191) handles double-click / Enter-while-highlighted.
- No models: a manual `Input` field with a hint to run `ollama pull`. `action_select` reads the input value.

Both paths dismiss with `None` on `escape`.

**`RuleEditorScreen` (`tui/screens/rule_editor.py`)**
A `ModalScreen[FolderRule | None]`. Five fields: name, target path, instructions, interval `Select`, allowed tools (one `Checkbox` per tool), and an enabled `Checkbox`. `_build_rule()` (line 232) validates all fields in order, updates the `#error-label` label, and returns `None` on validation failure to keep the modal open. On success it constructs a `FolderRule` with `uuid.uuid4()` as the ID for new rules, or preserves `rule.id` and `rule.created_at` for edits (line 259‚Äì271).

`_INTERVAL_OPTIONS` (line 21) is a `list[tuple[str, int]]` where the int is the raw hours value, which is what `Select` passes to `action_save` and `_build_rule` reads via `.value`.

**`MainScreen` (`tui/screens/main.py`)**
Three-pane layout: `TitleBar` docked top, a `Horizontal` containing `ChatWidget` (1fr width) and `SidebarWidget` (44 cells fixed), and `Footer` at the bottom. Added two methods:

```python
def on_screen_resume(self) -> None:
    self.call_after_refresh(self._restore_focus)

def _restore_focus(self) -> None:
    try:
        for inp in self.query("Input"):
            if not inp.disabled:
                inp.focus()
                return
    except Exception:
        pass
```

`_restore_focus` only calls `focus()` ‚Äî it does not call `set_input_enabled(True)`. This is deliberate: at `RUNNING` or `PLANNING` state the chat input is intentionally disabled; the method must not re-enable it.

**`StashApp` ‚Äî query scoping fixes and focus restoration (`tui/app.py`)**
All 11 sites in `StashApp` that called `self.query_one(...)` were changed to `self.screen.query_one(...)`. The root cause was Textual 8.x scoping: `App.query_one` searches only the app's own compose tree (the internal `_default` screen). `MainScreen` is a pushed screen, not in that tree. Every widget query from `StashApp` must go through `self.screen`.

New `_restore_main_focus` method (line 302):
```python
def _restore_main_focus(self) -> None:
    try:
        self.screen.query_one("ChatWidget").set_input_enabled(True)
    except Exception:
        pass
```

Called via `call_after_refresh` from two places: `_on_loading_done` (after the boot splash) and `_on_model_selected` (after the model picker closes). Both call sites are guarded to IDLE state only.

`on_mount` (line 135) uses `await self.push_screen(...)` ‚Äî the `await` ensures `LoadingScreen` is fully on the stack before `on_mount` returns.

**`health/ollama.py` ‚Äî fully implemented**
The three stub functions are now real:
- `fetch_models(endpoint)` (line 37) uses `ollama.AsyncClient.list()` and returns `[m.model for m in response.models]`.
- `check(endpoint, selected_model)` (line 44) wraps `fetch_models` in a `try/except` that raises `OllamaUnavailableError` on any connection failure. Returns `HealthResult` with the appropriate `HealthStatus` enum value.
- `pull_model(endpoint, model)` (line 91) calls `client.pull(model)`.

### Prototype debugging session

**Problem: TUI frozen after loading screen transition**
`prototype/demo.py` rendered and animated correctly, but after the `DemoLoadingScreen` dismissed, all user interaction was frozen ‚Äî no keyboard input, no bindings, no button clicks. The `ChatWidget` input field appeared but did not accept focus.

**Layer 1 ‚Äî focus discarded during active loading screen**
`ChatWidget.on_mount` calls `InputArea.set_enabled(True)` which calls `inp.focus()`. This fires while `DemoLoadingScreen` is already the active screen. Textual silently discards focus calls to widgets that are not on the active screen. Fix: defer focus restoration until after the screen transition completes using `call_after_refresh`.

**Layer 2 ‚Äî `App.query_one` cannot reach pushed screens (Textual 8.x)**
`DemoApp` (and `StashApp`) has no `compose()` method ‚Äî it has no widgets in its own tree. In Textual 8.x, `App.query_one(...)` searches `App._default_screen` only. Every call like `self.query_one(ChatWidget)` raised `NoMatches: No nodes match 'ChatWidget' on Screen(id='_default')`. The fix for all sites in both `DemoApp` and `StashApp` is `self.screen.query_one(...)`, where `self.screen` returns the currently active screen.

**Layer 3 ‚Äî unordered screen pushes**
`DemoApp.on_mount` called `self.push_screen(MainDemoScreen())` and `self.push_screen(DemoLoadingScreen())` without `await`. Without `await`, both pushes are scheduled asynchronously and may not complete in order. Changed to `await self.push_screen(...)` for both calls to guarantee ordering.

**Layer 4 ‚Äî terminal-level key collision**
`ctrl+m` was bound to the model picker action. At the terminal byte level, `ctrl+m` = ASCII 13 = `\r` = Enter. The binding was unreachable because Enter is consumed first. `ctrl+p` was tried as a replacement but is Textual's built-in command palette shortcut. Final binding: `ctrl+o` (not mapped by either the terminal or Textual internals).

### Answers to open questions from 2026-04-12

**1. Config key structure for `Agent`**
Resolved by reading the nested structure: `Agent.__init__` should read `config.get("ollama", {}).get("host", "http://localhost:11434")` and `config.get("ollama", {}).get("max_steps", 20)`. The current code at lines 105‚Äì107 of `core/agent.py` still uses flat keys (`config.get("ollama_host", ...)`, `config.get("max_steps", ...)`). This mismatch is a live bug ‚Äî the configured Ollama host and max_steps are never actually used; the agent always falls back to the hardcoded defaults. **Still open** ‚Äî fix is to update `Agent.__init__` to mirror the nested structure.

**2. `config["model"]` on first run**
The model picker flow prevents this in normal operation: `_on_loading_done` (line 139) checks `health_result.status` and pushes `ModelPickerScreen` before any `Agent` is constructed when `NO_MODEL_SELECTED` or `MODEL_MISSING`. After the picker dismisses, `_on_model_selected` writes the model to `self.config["ollama"]["model"]` in memory. `AgentFactory.build()` reads `config["model"]` ‚Äî this is still the flat key, not `config["ollama"]["model"]`, which means the model selection from the picker is also never passed to the agent correctly. **Linked to question 1** ‚Äî the config key flattening vs nesting inconsistency affects both host and model.

**3. `call_from_thread` inside async context**
Confirmed: after `await loop.run_in_executor(...)` returns in `_run_rule()`, execution is back on the event loop. Direct `self._app.post_message(...)` is correct here. `call_from_thread` is for calling from genuine non-async OS threads. The current usage is harmless in practice because Textual's `call_from_thread` implementation checks whether it's already on the event loop thread and handles it, but it is semantically wrong and should be replaced. **Still open** ‚Äî clean fix is two direct `post_message` calls after the executor returns.

**4. Canonical dependency declaration**
`pyproject.toml` exists and is the authoritative dependency file. `tomli_w` is declared there along with `textual`, `ollama`, `tinydb`, and `apscheduler`. No separate `requirements.txt`. **Resolved**.

**5. Plan ‚âÝ execution contract**
`Agent.plan()` and `Agent.run()` both call `_loop()` independently. Plan mode runs with real read-only tools (`ls`, `glob`) but skips write tools. Full run re-reasons from scratch with all tools enabled. The plan shown to the user is indicative, not binding ‚Äî the agent may take a different path during execution if real observations differ from plan-mode observations. This is acceptable for v1. The user approves the general intent and tool set, not a locked step sequence. **Accepted by design**.

**6. `ModelPickerScreen` and in-memory config update**
Resolved. `_on_model_selected` (line 293 of `tui/app.py`) calls `self.config.setdefault("ollama", {})["model"] = model` before calling `_save_config()`. This updates the in-memory dict that `AgentFactory` reads at `build()` time. No restart required ‚Äî the next agent construction picks up the new model. `_save_config()` also writes to `config.toml` via `tomli_w`. **Resolved**.

### Decisions taken

**`PaneHeader` not extracted to a shared module**
Both `ChatWidget` and `SidebarWidget` define their own `PaneHeader` class with identical structure. The decision was to leave them co-located with their respective files rather than creating a shared `widgets/common.py`. The classes are six lines of logic; the overhead of an import indirection exceeds the benefit of deduplication at this scale.

**`display: none` over `visible: hidden` for `ApproveBar` and badges**
`display: none` removes the widget from layout flow ‚Äî other widgets fill the space. `visible: hidden` hides the widget but preserves its space. Collapsing the approve bar and the model/rules badges entirely is the correct visual behaviour; a fixed-height gap when they're hidden would be wrong.

**`RichLog` over `ListView` or manual `ScrollableContainer` for the audit log**
`RichLog` supports Rich markup natively, handles auto-scroll, and is optimised for append-only streaming. The audit log is exactly this use case ‚Äî timestamped tool results appended live. A `ScrollableContainer` with mounted `Label` children would work but would accumulate DOM nodes indefinitely; `RichLog` manages its own internal buffer.

**`on_screen_resume` focus restoration uses `query("Input")` not `query_one(InputArea)`**
`_restore_focus` on `MainScreen` uses `self.query("Input")` and takes the first non-disabled one. This is intentionally permissive ‚Äî it works regardless of which modal just closed or which run state is active. A `query_one(InputArea)` approach would require importing `InputArea` into `MainScreen`, creating a cross-layer dependency. The CSS-string query keeps `MainScreen` layer-agnostic.

**`LoadingScreen` uses `dismiss()` not `pop_screen()`**
`Screen.dismiss(value)` both pops the screen and delivers the typed return value to the callback registered in `push_screen(screen, callback)`. `pop_screen()` has no return value mechanism. Since `LoadingScreen` needs to pass the pre-computed `HealthResult` back to `StashApp._on_loading_done`, `dismiss` is the only correct API.

**`FolderRulesSection.load_rules` is a full replace, not a diff**
When `StashApp._on_rule_saved` calls `load_rules(rules_db.all())`, all existing `RuleItem` children are removed and remounted. A diff/patch approach would be more efficient but adds complexity for a list that rarely has more than 10‚Äì20 entries. Full replace is simpler and correct.

Status: **open** ‚Äî fix is to update `Agent.__init__` to use nested key access.

`StashScheduler._run_rule()` calls `self._app.call_from_thread(self._app.post_message, ...)` at lines 130 and 136 while already on the event loop thread. Status: **open** ‚Äî replace with direct `post_message` calls.
