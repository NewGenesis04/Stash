### What was built

**Persistence layer â€” SQLite (`persistence/sqlite.py`)**
The `_migrate()` function creates three tables (`task_runs`, `react_steps`, `conversations`) and enables WAL mode via `PRAGMA journal_mode=WAL` at line 31. Five supporting functions were implemented: `begin_run()` (line 64), `finish_run()` (line 73), `log_step()` (line 82), `add_message()` (line 102), and `get_history()` (line 111). `get_history` uses `IS ?` rather than `= ?` for the `rule_id` parameter so that `NULL` comparisons work correctly when querying the manual chat session.

**Persistence layer â€” TinyDB (`persistence/tinydb.py`)**
The `RulesDB` class had five stub methods filled in: `all()` (line 36), `get()` (line 39), `upsert()` (line 44), `delete()` (line 49), and `update_last_run()` (line 54). All five use TinyDB's `Query()` pattern. `upsert` stores rules via `rule.model_dump()`, keeping the JSON file human-readable and consistent with the `FolderRule` Pydantic model (line 18).

**Callback chain (`core/callbacks.py`)**
`AuditLogger` (line 27) was implemented to write every tool call result to SQLite via `db.log_step()` in `on_after`, and to call `db.finish_run()` with `"failed"` in `on_error`. `StatusTracker` (line 75) was simplified to a single responsibility: `on_error` calls `rules_db.update_last_run()` with `"failed"`. Success marking is the scheduler's job, not a per-tool concern. `TUIUpdater` (line 53) was already implemented; its `on_after` posts a `ReactStepReady` message to `StashApp` via `call_from_thread`.

**Agent core (`core/agent.py`)**
The full ReAct loop was implemented across several components:

- `_parse_response()` (line 53) parses raw model output using `match` on `line.strip().split(":", 1)` â€” destructuring each line into a `[key, value]` pair. This handles `"Action Input"` vs `"Action"` without ambiguity since the full left-hand side of the first colon is the key.
- `Agent._loop()` (line 119) is the core loop. It builds the message history, calls `ollama.Client.chat()` on each iteration, parses the response, appends steps, and fires callbacks via `_call_tool()`. The `for/else` construct at line 182 catches `max_steps` exhaustion without a separate flag.
- `Agent._call_tool()` (line 190) is the single point where all three callbacks fire. It re-raises `UnauthorisedToolError` explicitly before the generic `except Exception` block, preventing the error from being softened into an observation string.
- `Agent.plan()` (line 109) and `Agent.run()` (line 113) both delegate to `_loop()` with `dry_run=True` and `dry_run=False` respectively. Both accept an optional `run_id` parameter for log correlation.
- `AgentFactory` (line 214) was added to `core/agent.py`. Its `build()` method (line 225) constructs an `Agent` and a full-access `SessionRegistry`, accepting a SQLite connection and callback list from the caller. This keeps `StashApp` from needing to import tool internals directly.

**Readonly-aware dry-run (`core/agent.py`, `tools/ls.py`, `tools/glob.py`)**
`ls` and `glob` schemas were given a `"readonly": True` field. In `Agent._loop()` at line 158, each tool call checks this flag: read-only tools execute for real in plan mode, write tools return `"[plan mode â€” not executed]"`. This ensures the plan the user approves is grounded in real filesystem state rather than hallucinated observations.

**Structured logging**
`core/agent.py` logs at `INFO` for every ReAct step â€” `agent.thought`, `agent.action`, `agent.observation`, `agent.final` â€” and at `ERROR` for parse failures and unauthorised tool attempts. All log entries carry a `run_id` context dict for correlation. `persistence/sqlite.py` logs at `DEBUG` for each DB write. `persistence/tinydb.py` logs at `DEBUG` for CRUD operations. `core/callbacks.py` logs at `DEBUG` in `AuditLogger.on_after` and `ERROR` in both `AuditLogger.on_error` and `StatusTracker.on_error`. The `JsonFormatter` in `stash/log.py` (pre-existing) serialises all entries including arbitrary `extra` fields.

**Scheduler (`scheduler/runner.py`)**
`StashScheduler` was fully implemented. `_register()` (line 82) adds interval jobs with `max_instances=1` and `replace_existing=True`. `_run_rule()` (line 94) is `async def` â€” it opens a fresh SQLite connection per run, builds `AuditLogger`, `TUIUpdater`, and `StatusTracker` callbacks, constructs an `Agent` directly (not via `AgentFactory`), and runs `agent.run()` via `asyncio.get_event_loop().run_in_executor()` to avoid blocking the event loop. The connection is closed in a `finally` block (line 138). `reschedule()` (line 60) removes the existing job if present before re-registering. `remove()` (line 72) warns if no job is found rather than silently succeeding.

**App orchestration (`tui/app.py`)**
`StashApp` was rebuilt with a four-state explicit state machine (`RunState` enum, line 88): `IDLE â†’ PLANNING â†’ AWAITING_APPROVAL â†’ RUNNING â†’ IDLE`. State transitions happen in `on_task_submitted()` (line 189), `on_plan_approved()` (line 219), and `on_plan_rejected()` (line 244). Between `AWAITING_APPROVAL` and `RUNNING`, a `PendingRun` Pydantic model (line 95) holds the `run_id`, `Agent`, `SessionRegistry`, and task string. Three new cross-layer message types were added: `TaskSubmitted`, `PlanApproved`, `PlanRejected` (lines 67â€“81). The three pre-existing stubs â€” `on_react_step_ready`, `on_rule_completed`, `on_ollama_status_changed` â€” were implemented as single-line delegations to their respective widgets via CSS string queries.

**Entrypoint (`main.py`)**
`main()` (line 61) implements a ten-step boot sequence: config load with first-run default generation, logging setup, data directory creation, SQLite connection, TinyDB open, health check via `asyncio.run()`, object graph construction, `StashApp` construction, circular dependency wiring via `scheduler.set_app(app)` (line 119), and `app.run()`. `load_config()` (line 48) writes `_DEFAULT_CONFIG_TOML` to disk if no config file exists. `_hard_exit()` (line 56) prints to stderr and calls `sys.exit(1)` for the `OllamaUnavailableError` case.

**Prototype (`prototype/models.py`)**
`ReActStep` was converted from a `dataclass` to a Pydantic `BaseModel` (line 7). `datetime.utcnow()` was replaced with `datetime.now(UTC)` â€” the former is deprecated since Python 3.12 and returns a naĂŻve datetime. `prototype/app.py` was unaffected; it accesses `ReActStep` fields as attributes, which Pydantic exposes identically to dataclasses.

### Decisions taken

**`UnauthorisedToolError` as a hard abort, not a soft observation**
Initially, `_call_tool()` caught all exceptions uniformly and returned them as error strings â€” meaning an unauthorised tool attempt would appear to the model as an observation, giving it the opportunity to try again with a different tool. The decision was made to re-raise `UnauthorisedToolError` specifically at `core/agent.py:198`, catch it explicitly in `_loop()` at line 168, append an `error` step, and `break`. The model gets no second chance. The `SessionRegistry.call()` boundary in `core/registry.py:22` is now a genuine hard stop.

**Readonly-aware plan mode**
The original dry-run implementation returned `"[plan mode â€” not executed]"` for all tool calls, including `ls` and `glob`. This meant the agent planned against hallucinated observations â€” it had no idea what files actually existed. The alternative considered was a two-phase approach (exploration phase with real tools, then write-phase with approval). That was rejected as overly complex. The chosen solution adds a `"readonly": True` field to the `ls` and `glob` schemas and checks it in `_loop()` at line 159. Read-only tools execute for real in plan mode; write tools return the placeholder. Plans are now grounded in actual filesystem state.

**`AgentFactory` as an injectable**
Rather than constructing `Agent` instances inline in `StashApp` with raw tool dependencies, `AgentFactory` (line 214 of `core/agent.py`) encapsulates construction. `StashApp` receives a factory instance and calls `build()`, which returns an `(Agent, SessionRegistry)` tuple. This keeps the app constructor clean and centralises agent configuration. The scheduler does not use `AgentFactory` â€” it constructs `Agent` directly in `_run_rule()` because it needs finer control over the callback set and config keys.

**`PendingRun` as a Pydantic model with `arbitrary_types_allowed`**
The state held between plan approval and execution â€” `run_id`, `Agent`, `SessionRegistry`, `task` â€” needed a container. A `dataclass` was considered and rejected in favour of a Pydantic `BaseModel` with `ConfigDict(arbitrary_types_allowed=True)` (line 97). This is consistent with the project-wide preference for Pydantic over dataclasses.

**Deferred `StashApp` wiring via `scheduler.set_app()`**
`StashScheduler` needs a reference to `StashApp` to post messages. `StashApp` needs a reference to `StashScheduler` to start it. Neither can be constructed first. The chosen resolution was to remove `app` from `StashScheduler.__init__` and add `set_app()` (line 45 of `scheduler/runner.py`). `main.py` constructs both, then calls `scheduler.set_app(app)` at line 119 before `app.run()`. An `assert self._app is not None` guard in both `start()` (line 49) and `_run_rule()` (line 95) enforces the contract.

**Per-run SQLite connections in the scheduler**
The main app holds a single SQLite connection for the chat session. Scheduler jobs run in a thread pool executor. Sharing one connection across threads without careful locking risks corruption. The decision was to open a fresh `db.connect()` call at the start of each `_run_rule()` invocation (line 106) and close it in `finally` (line 138). WAL mode (enabled in `_migrate()` at line 31) allows the main connection and per-run connections to coexist safely.

**`max_instances=1` on all scheduler jobs**
APScheduler's default behaviour is to allow concurrent runs of the same job if the interval fires while a previous run is still active. For a file agent this is dangerous â€” two concurrent runs against the same folder can produce conflicting operations. `max_instances=1` is set in `_register()` at line 89.

**Sequential `asyncio.run()` then `app.run()` for the health check**
`health.check()` is `async def`. It must be called before Textual launches. `asyncio.run()` was used in `main.py` at line 87 for just this step. Textual's `app.run()` starts its own event loop internally. These are sequential calls â€” there is no nesting.

### Problems identified

**Config key path mismatch between `main.py` and `Agent.__init__`**
`Agent.__init__` at line 105 reads `config.get("ollama_host", "http://localhost:11434")`. The config dict produced by `main.py` has structure `{"ollama": {"host": "..."}}` â€” there is no top-level `"ollama_host"` key. The configured host is therefore never used; the agent always falls back to the hardcoded default. Similarly, `Agent.__init__` at line 107 reads `config.get("max_steps", 20)`, but `max_steps` lives at `config["ollama"]["max_steps"]` in the TOML structure. Status: **open**.

**`config["model"]` raises `KeyError` on first run**
`Agent.__init__` at line 106 reads `config["model"]` without a `.get()` fallback. On first run, before the model picker has written a selection to `config.toml`, the key does not exist. Any attempt to construct an `Agent` before model selection would raise `KeyError`. The model picker flow in `on_mount` prevents this in normal usage, but the absence of a safe fallback is fragile. Status: **open**.

**`call_from_thread` used inside an async context in the scheduler**
`StashScheduler._run_rule()` is an `async def` running in the event loop. After `await loop.run_in_executor(...)` returns, execution is back on the event loop thread. At lines 130 and 136, `self._app.call_from_thread(self._app.post_message, ...)` is called â€” but `call_from_thread` is intended for use from non-async threads to safely schedule a call on the event loop. Using it from within the event loop is redundant and potentially incorrect; `self._app.post_message(...)` directly would be correct here. Status: **open**.

**`tomli_w` is an undeclared dependency**
`main.py` imports `tomli_w` at line 23 for writing the default config to disk. No `pyproject.toml` or `requirements.txt` was found in the repository. `tomli_w` is not part of the stdlib and will cause an `ImportError` if not installed. Status: **open** (dependency list not yet established).

**`health.check()`, `health.fetch_models()`, `health.pull_model()` are stubs**
All three functions in `health/ollama.py` raise `NotImplementedError`. `main.py` and `StashApp._poll_ollama()` both handle this gracefully with a `try/except NotImplementedError` block, but Ollama health is entirely non-functional until these are implemented. Status: **open**.

**TUI widgets are unimplemented**
`StashApp` calls `self.query_one("ChatWidget")`, `self.query_one("Sidebar")`, and `self.query_one("TitleBar")` in several handlers. None of these widgets exist yet. These calls will raise `NoMatches` at runtime. The app cannot be launched in a usable state. Status: **open** (by design â€” TUI layer is next).

### Open questions

1. What is the correct config key structure for `Agent`? Should `Agent.__init__` read `config["ollama"]["host"]` and `config["ollama"]["max_steps"]`, or should `main.py` flatten the config before passing it down?
2. Should `config["model"]` access in `Agent.__init__` be a `.get()` with a `None` fallback, or should construction be prevented entirely if no model is set?
3. Is `call_from_thread` the correct Textual API to use for posting messages from within an `async def` that is already on the event loop thread, or should direct `post_message` be used after `run_in_executor` returns?
4. What is the canonical dependency declaration file for this project â€” `pyproject.toml`, `requirements.txt`, or both?
5. The plan produced by `Agent.plan()` is not a guaranteed execution contract â€” `Agent.run()` re-reasons from scratch with real observations and may take a different action sequence than the plan showed. Is this acceptable for the v1 scope, or does it need a closer relationship between plan and execution?
6. When `ModelPickerScreen` writes a selected model back to `config.toml` via `tomli_w`, does it also need to update `StashApp.config` in memory so that subsequent `Agent` constructions see the new value without a restart?
