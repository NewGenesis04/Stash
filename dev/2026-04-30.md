## 2026-04-30 ‚Äî Location registry: grounding the agent in real filesystem paths

### What was built

This session addressed a fundamental correctness problem: the agent had no mechanism for resolving informal folder references ("my downloads folder", "the movies folder") to real filesystem paths. Without it, the model either hallucinated paths that do not exist, constructed plausible-sounding but wrong ones, or ‚Äî as observed in a recorded exchange ‚Äî fabricated file names and returned them as if it had actually read the directory.

The solution is a location registry: a persistent store of user-declared folder entries, each with a canonical name, a list of aliases, and an absolute path. The agent queries this registry via a dedicated tool on every folder reference. If the name is not found, a directory picker fires and the result is stored before the agent continues.

---

**`stash/persistence/tinydb.py` ‚Äî `LocationEntry`, `LocationsDB`**

`LocationEntry` (lines 18‚Äì23) is a Pydantic model with fields `name`, `aliases` (list), `path`, `added`, and `last_verified`, all as ISO timestamp strings. `LocationsDB` (lines 26‚Äì59) opens a separate TinyDB instance backed by `~/.stash/locations.json`. Key methods:

- `resolve(name)` (line 34): iterates all documents, matching on canonical name or any alias, case-insensitively. Returns the first `LocationEntry` hit or `None`.
- `upsert(entry)` (line 44): keyed on `name` via TinyDB's `Query().name == entry.name`.
- `delete(name)` (line 49): removes by name.
- `verify(entry)` (line 54): checks `Path(entry.path).exists()`, updates `last_verified` on the stored document if the path is live, and returns a `(bool, LocationEntry)` tuple.

`LocationsDB` is structurally parallel to the existing `RulesDB` (line 75) ‚Äî same file, same TinyDB pattern, separate table name. Using a separate JSON file (`locations.json` vs `rules.json`) keeps the two stores independent.

**`stash/tools/resolve_location.py` ‚Äî `make_resolve_location_tool()`**

`ResolveLocationArgs` (line 13) is the Pydantic validator: a single `name: str` field. The tool schema (lines 17‚Äì29) is marked `"readonly": True`, meaning it executes during both the plan phase and the run phase.

`make_resolve_location_tool()` (lines 32‚Äì51) is a factory function rather than a module-level callable. It takes a `LocationsDB` instance and a `request_picker: Callable[[str], str | None]` and returns the actual tool function as a closure. This is necessary because the picker callback must reference the live `StashApp` instance, which does not exist at import time.

The returned `resolve_location(name)` function:
1. Calls `locations_db.resolve(name)`.
2. If found: verifies the path still exists. If it does not, returns an error string instructing the agent to ask the user to re-register.
3. If not found: calls `request_picker(name)`, which blocks until the user completes or cancels the picker. Returns the path on success, or an error string if the user cancelled.

The tool never returns a path it has not verified exists on disk.

**`stash/tui/app.py` ‚Äî threading mechanism and new binding**

`StashApp.__init__` (line 110) gains a `locations_db: LocationsDB` parameter (line 116), stored as `self._locations_db`.

`request_location(name)` (line 369) is the method passed to `_picker_cell` in `main.py`. It is called from the executor thread (the agent's `run_in_executor` pool) and must block that thread while the TUI handles the picker interaction. The mechanism:

1. Creates a `threading.Event` and a single-element mutable list `result: list[str | None] = [None]`.
2. Defines a local `on_picked(entry)` callback that writes to `result[0]` and calls `event.set()`.
3. Calls `self.call_from_thread(self._open_location_picker, name, on_picked)`, which schedules `_open_location_picker` on the asyncio event loop thread without blocking.
4. Calls `event.wait()`, blocking the executor thread.
5. When the picker is dismissed, Textual calls `on_picked` on the event loop thread, setting the event.
6. The executor thread unblocks and returns `result[0]`.

`_open_location_picker(name, callback)` (line 384) simply calls `self.push_screen(LocationPickerScreen(suggested_name=name), callback)`. The agent sees `resolve_location` as a tool call that returned a path; it has no visibility into the UI interaction.

`action_open_locations()` (line 363) handles the manual `ctrl+p` path, pushing `LocationRegistryScreen` directly.

A `ctrl+p` binding was added to `StashApp.BINDINGS` (line 106).

### Decisions taken

**User declaration over inference**

The original problem had several potential solutions: OS-level path resolution (using `Path.home() / "Downloads"` for common folders), fuzzy name matching against directory listings, or an LLM-judged similarity pass. All of these involve the agent making decisions about what a name means.

The approach chosen is the opposite: the agent never decides. Every path in the registry was put there by the user, either proactively or in response to a specific unresolved reference. The agent cannot be wrong about a registered location because the OS returned the path; the user confirmed it. Fuzzy matching and inference were rejected on correctness grounds ‚Äî they trade occasional convenience for occasional silent misdirection.

**Aliases as vocabulary, not fuzzy matching**

Aliases are stored as an explicit list on the entry. Resolution is exact string equality after lowercasing and stripping. The user teaches the agent their vocabulary incrementally ("films" maps to Movies, "dl" maps to Downloads). This is more predictable than edit-distance matching and requires no threshold tuning. The cost is that an alias must be registered before it works; the benefit is that it always works once it is.

**`make_resolve_location_tool()` factory over a class with `__call__`**

The tool registry accepts plain callables. Injecting dependencies (the `LocationsDB` instance and the picker callback) required either a factory function producing a closure, or a class with `__call__`. A closure was chosen as the simpler option, consistent with how the rest of the tool set is structured (plain functions).

**Mutable cell for the picker callback in `main.py`**

The picker callback must reference `app.request_location`, but `app` does not exist when the tool is constructed. Three options were considered:

1. Build the tool after `app`, passing `app.request_location` directly.
2. Use a mutable cell to defer binding.
3. Add a `set_picker(callback)` method to the tool.

Option 1 would require `ToolRegistry`, `AgentFactory`, and `StashScheduler` to be constructed after `app`, inverting the current dependency order. Option 3 adds public API to a tool that has no other reason for it. Option 2 (the mutable cell) keeps the existing construction order intact and mirrors the `scheduler.set_app(app)` pattern already in the codebase.

**`threading.Event` for cross-thread blocking**

The agent runs inside `loop.run_in_executor(None, ...)` ‚Äî a `ThreadPoolExecutor` thread. The TUI runs on the asyncio event loop. The picker interaction must complete before `resolve_location` can return a path. `threading.Event` is the standard synchronisation primitive for this pattern: the executor thread blocks on `event.wait()`, the asyncio thread calls `event.set()` from the picker dismiss callback. No asyncio primitives were appropriate here because `asyncio.Event` is not thread-safe and would require `loop.call_soon_threadsafe`.

**Separate `locations.json` file, not a new table in `rules.json`**

`RulesDB` opens its own `TinyDB` instance pointed at `rules.json`. The simplest option for locations would have been to add a second table to the same TinyDB instance. This was rejected because `RulesDB` does not expose its `TinyDB` instance ‚Äî adding a second consumer would require modifying `RulesDB`'s constructor or leaking the internal `_db` attribute. A separate `locations.json` file keeps the two stores fully independent and follows the existing pattern.

**`resolve_location` marked `readonly: True`**

The tool schema includes `"readonly": True` (line 28 of `resolve_location.py`). In the agent loop (`stash/core/agent.py`, line 108), read-only tools are executed during plan mode (`dry_run=True`); non-read-only tools are skipped. Marking `resolve_location` as read-only means the planner receives real resolved paths, producing a meaningful plan with concrete filesystem paths rather than placeholder strings. The side-effect ‚Äî that an unknown location triggers the picker during planning, before the user has approved the plan ‚Äî was considered acceptable: the user registers the location once, it persists, and all subsequent calls hit the registry immediately.

### Problems identified

**`glob_tool` function default inconsistent with `GlobArgs` model default ‚Äî resolved**

`GlobArgs.base_path` was changed to `"~"` in the model but `glob_tool`'s function signature still defaulted to `"."`. When called through the tool registry (via Pydantic validation), the model's default was used (`"~"`). When called directly ‚Äî as in the test suite ‚Äî the function's own default was used (`"."`). The two code paths produced different behaviour for the same omitted argument. Fixed by updating the function signature at `stash/tools/glob.py:26` to match the model default.

**`test_default_base_path_uses_cwd` testing superseded behaviour ‚Äî resolved**

`stash/tests/test_tools.py` contained a test that called `glob_tool("*.txt")` with no `base_path`, changed the working directory to a `tmp_path` fixture via `monkeypatch.chdir`, and asserted that files in that directory were found. This test was specifically validating the old `"."` default and would continue to pass after the function signature fix (since the function still defaulted to `"."` before the fix), but tested intentionally discarded behaviour. The test was replaced with `test_default_base_path_uses_home` (line 236 of the updated test file), which verifies that the default search root contains the user's home path.

**Edit flow creates duplicate registry entry when name is changed ‚Äî resolved**

In `LocationRegistryScreen.on_button_pressed`, the Edit path originally used `_on_entry_saved` as its dismiss callback. That callback simply calls `locations_db.upsert(entry)`, which keys on `entry.name`. If the user changed the name (e.g. "Movies" ‚Üí "Films"), `upsert` inserted a new "Films" entry without removing the original "Movies" entry. Both entries then existed in the registry.

Fixed by replacing the shared `_on_entry_saved` callback in the Edit path with a closure (`_on_edited`, lines 153‚Äì160 of `location_registry.py`) that captures `original_name = entry.name` before pushing the picker. On dismiss, if `updated.name != original_name`, it calls `locations_db.delete(original_name)` before upserting the new entry.

### Open questions

1. **Should `resolve_location` fire the picker during the plan phase?** It is currently marked `readonly: True`, so it executes during dry-run planning. This means an unknown location triggers the picker before the user has seen or approved the plan. The user registers the location, planning continues, and the plan shows real paths ‚Äî which is arguably better than placeholder strings. But the user interaction happens at an unexpected moment. Should `resolve_location` be marked non-readonly so it is skipped in planning (returning `"[plan mode ‚Äî not executed]"`) and only fires in the execution phase?

2. **How should the agent handle a path that was registered but has since been deleted or moved?** Currently `resolve_location` returns an error string instructing the agent to ask the user to re-register. There is no automatic trigger to re-open the picker. Should the error string include a signal that the registry UI can intercept, similar to how the "not found" case works?

3. **Should the registry be pre-populated on first run with standard directories that exist on disk?** The current design grows the registry lazily as the agent encounters unknown names. A first-run setup pass that resolved `~/Downloads`, `~/Documents`, `~/Desktop`, etc. and offered them as pre-checked options would reduce the number of picker interruptions on early sessions. This was discussed and deferred; the lazy flow is sufficient.

4. **Should `resolve_location` be available to scheduled rules?** It is currently excluded from `_ALL_TOOLS` in `stash/tui/screens/rule_editor.py` (line 19). A scheduled rule that references a folder by name would fail silently. The counter-argument is that scheduled rules run unattended and the picker cannot fire without a user present. If rules need location resolution, they should be authored with absolute paths. Is this the right constraint, or should rules resolve locations at registration time?

5. **What happens if `event.wait()` in `request_location` never returns?** If the picker is somehow dismissed without calling the callback (e.g. the screen is forcibly removed from the stack by an unhandled exception), `event.wait()` blocks the executor thread indefinitely. There is no timeout. Should `event.wait(timeout=...)` be added, and if so, what is the appropriate fallback?

### Decisions taken

**`FolderPickerScreen` as a separate class rather than reusing `LocationPickerScreen`**

`LocationPickerScreen` returns a `LocationEntry` ‚Äî it requires the user to supply a name and optional aliases alongside the path. Using it for the rule editor's target path field would force the user to fill in registration metadata just to populate a single form field. The interface contract is wrong for the use case.

A new `FolderPickerScreen` was created that returns `str | None` ‚Äî just the path. It has no name or aliases inputs. The two screens share visual structure (same `DirectoryTree` + label + footer pattern) but serve different purposes and have different return types. Reuse via inheritance or composition was not pursued; the duplication is at the CSS level only, and keeping the classes independent avoids coupling a simple picker to the registration flow.

**Editable Input plus Browse, not picker-only**

An alternative was to replace the `#inp-path` Input with a non-editable display label, making the directory picker the only way to set the path. This was rejected: it removes the ability to paste an absolute path directly, which is a common action for developers who know their paths. The Input remains fully editable; Browse is an additive affordance.

**Prototype imports production `RuleEditorScreen`**

The prototype previously maintained a local duplicate of `RuleEditorScreen` that operated over `dict` rather than `FolderRule`. This meant the demo diverged from the production screen over time, and any new feature added to the production screen (like Browse) would not appear in the demo without a manual parallel change. Switching the prototype to import the real class means the demo is always accurate and Browse is automatically present.

The cost is that the prototype now depends on the full `stash` package, including Pydantic models and persistence code. This was considered acceptable ‚Äî the prototype directory already imports from `stash` in `location_picker.py`.

**120-second timeout for `event.wait`**

The timeout is a safety net against the callback never firing ‚Äî not a UX constraint on the user. Normal picker interactions complete in seconds. The timeout must be long enough that a slow user navigating a large directory tree on a sluggish machine does not hit it; 120 seconds provides a comfortable margin over any realistic interaction. On timeout, the fallback is to return `None` from `request_location`, which the `resolve_location` closure already handles by returning an error string to the agent.

### Problems identified

**`action_select_folder` is a silent no-op when no folder has been selected ‚Äî open**

`FolderPickerScreen.action_select_folder` (line 106) checks `if self._selected_path` before dismissing. If the user presses `ctrl+s` or clicks ‚úì Select without having navigated the `DirectoryTree`, nothing happens ‚Äî no error message, no visual feedback. The same applies via the button route, which calls `action_select_folder` directly. A user unfamiliar with the screen would receive no signal explaining why the action did not work. An error label (matching the pattern in `LocationPickerScreen.action_save`, line 154) or a brief label flash would resolve this.

**`prototype/location_picker.py` accidentally modified then reverted ‚Äî resolved**

During an intermediate iteration, `FolderPickerScreen` was imported into `prototype/location_picker.py` and an `f` binding was added to `_BgScreen`. This was incorrect ‚Äî `location_picker.py` is a self-contained demo for `LocationPickerScreen` and `LocationRegistryScreen` and should not reference the folder picker. The file was restored to its pre-session state. The folder picker demo belongs in `prototype/rule_editor.py`, which now demonstrates the full rule editor flow including Browse.
