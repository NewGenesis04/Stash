# Stash Devlog

## 2026-05-07 — Context safety, plan caching, and dynamic examples implemented; TUI test suite fixed

### What was built

**Plan caching to eliminate duplicate Ollama round-trips**

`Agent._loop()` (`stash/core/agent.py`, line 68) was changed from returning `list[ReActStep]` to returning `tuple[list[ReActStep], list[dict]]` — the second element being the full `messages[]` array as it stood at the end of the planning loop. `Agent.plan()` (line 57) surfaces this tuple directly. `Agent.run()` (line 61) gained an `initial_messages: list[dict] | None` parameter; when provided, `_loop()` skips the full context assembly and resumes from the cached state (lines 76–83).

`PendingRun` in `stash/tui/app.py` (line 88) gained a `messages: list[dict]` field to carry the cached messages between plan approval and execution. `on_task_submitted` (line 206) unpacks the tuple returned by `agent.plan()`. `on_plan_approved` (line 250) passes `pending.messages` into `agent.run()`.

**Non-blocking plan phase for `resolve_location`**

`StashApp.request_location()` (`stash/tui/app.py`, lines 373–374) now checks whether the app is in `RunState.PLANNING` and returns a placeholder string — `"[plan mode — will prompt user to pick folder for '{name}']"` — instead of blocking the executor thread on the folder picker UI. The picker is deferred to the execute phase.

**Result capping on `ls` and `glob`**

`ls_tool()` (`stash/tools/ls.py`, lines 47–54) caps its output at 100 entries and appends `\n... and N more` when the directory contains more. `glob_tool()` (`stash/tools/glob.py`, lines 51–58) applies the same cap to match results.

**Dynamic few-shot examples**

A new module `stash/prompts/examples.py` exports `get_examples(home_path: str)` (line 6), which renders the four standard few-shot traces with the user's actual home directory substituted in for all paths. `build_system_prompt()` (`stash/prompts/prompt.py`, lines 74–85) now calls `get_examples()` at assembly time and injects the result as a `## Examples` block. The previously hardcoded `## Examples` section — which contained literal `C:\Users\NewGenesis04` paths — was removed from `SYSTEM_PROMPT`.

**Context pressure warning and preference cap**

`Agent._loop()` (lines 88–97) estimates token count at each iteration via `total_chars // 4` and logs a `agent.context_pressure` warning if the estimate exceeds 80% of `context_window` (configurable via `config["ollama"]["context_window"]`, defaulting to 32768). `Agent._load_preferences()` (lines 183–186) truncates the preferences file content at 2,000 characters and appends a notice if the limit is exceeded. `Agent._load_session_history()` (lines 195–196) reads a `history_limit` key from `config["ollama"]`, defaulting to 20, rather than hardcoding the limit at the call site.

**SQLite session scoping fix**

`get_history()` (`stash/persistence/sqlite.py`, lines 137–143) now returns early with an empty list and a warning log if `session_id` is `None` when querying interactive chat history. The SQL operator was changed from `IS ?` to `= ?` so that a `None` slipping past the guard cannot accidentally match all rows where `session_id IS NULL`.

**TUI test suite**

`stash/tests/test_agent_loop.py` was updated to unpack the new tuple return from `agent.plan()` (e.g. `steps, _ = agent.plan(...)`). `stash/tests/test_tui.py` was updated so `mock_agent_factory` returns `agent.plan` as a `MagicMock` returning `(_PLAN_STEPS, [])` (line 93) to match the new signature.

Two bugs in the test infrastructure were identified and fixed (see Problems below).

### Decisions taken

**Returning `messages[]` from `_loop()` rather than caching it on the `Agent` instance**

The alternative would have been to store the plan-phase messages as instance state on `Agent` and have `run()` read it implicitly. This was rejected because `Agent` is documented as stateless per run — storing intermediate state on the instance would make concurrent use (e.g. two rules running simultaneously via `APScheduler`) unsafe. Threading the messages through `PendingRun` keeps `Agent` stateless and makes the data flow explicit.

**Placeholder string rather than a flag for plan-mode `resolve_location`**

The alternative was to pass a `dry_run` boolean into the tool functions themselves. This would require either changing the tool function signatures or threading the flag through `SessionRegistry`. Instead, `StashApp.request_location()` checks `self._run_state` directly — a single-line guard that requires no changes to the tool layer or registry. The downside is that `request_location` now depends on app state, which couples the tool's interactive fallback to the TUI. This was accepted as a pragmatic trade-off given that `request_location` was already a TUI-level method rather than a pure tool function.

**Character-count heuristic for token estimation**

No Ollama API exists to query the token count of a message list before sending it. The `total_chars // 4` heuristic is an approximation; actual token counts vary by model and content. A `context_window` config key was introduced to allow per-model tuning, but no models are pre-populated with values. The warning is advisory only — the loop does not halt or truncate on context pressure.

**Restricting anyio to asyncio in `conftest.py`**

`pytest-anyio` runs `@pytest.mark.anyio` tests against all installed backends. `trio` is not installed in this project (it is not listed in `pyproject.toml`'s `[dependency-groups]`), causing every test to be collected and immediately fail with `ModuleNotFoundError`. A root-level `conftest.py` was added that overrides the `anyio_backend` fixture to `["asyncio"]` only. The alternative — adding `trio` as a dev dependency — was not warranted since the project does not use trio and Textual's test runner runs on asyncio.

### Problems identified

**`RuleItem` widget ID used dashes; `update_rule_status` queried with underscores**

`RuleItem.__init__` (`stash/tui/widgets/sidebar.py`, line 117) was setting `id=f"rule-{rule.id}"`, producing IDs such as `rule-rule-99`. `FolderRulesSection.update_rule_status()` (line 156) computed `safe_id = f"rule-{rule_id}".replace("-", "_")` — producing `rule_rule_99` — before querying. The two strings never matched. The `query_one()` call silently failed (caught by a bare `except Exception`) and the status dot was never updated.

**Status:** Resolved. `RuleItem.__init__` now applies the same `.replace("-", "_")` normalisation (line 117), making the stored ID consistent with the query.

**`_test_on_mount` called `load_rules` before `MainScreen` was composed**

The test fixture in `stash/tests/test_tui.py` (inside `make_app`) monkeypatches `StashApp.on_mount` to push `MainScreen` and immediately call `self.screen.query_one("SidebarWidget").load_rules(...)`. In Textual, `push_screen()` schedules the screen's mounting asynchronously — the screen's child widgets are not yet in the DOM at the point the call returns. The `query_one()` raised `NoMatches`, which was caught by a `try/except: pass`, silently leaving the sidebar with no rule items loaded. The subsequent assertion in `test_rule_completed_updates_sidebar` — `pilot.app.screen.query_one("#rule_rule_99")` — failed with `NoMatches`.

**Status:** Resolved. `_test_on_mount` now calls `self.call_after_refresh(_load_rules)` (lines 120–126), deferring the `load_rules` call to after the next render cycle, by which point `MainScreen`'s widget tree is fully composed.

**All `[trio]` variants of every TUI test failing**

`pytest-anyio` collected 28 tests (14 × 2 backends) despite `trio` not being installed, producing 14 `ModuleNotFoundError: No module named 'trio'` failures.

**Status:** Resolved. `conftest.py` at the project root overrides the `anyio_backend` fixture to `["asyncio"]`, reducing collection to 14 tests, all of which pass.

**Token estimation is a heuristic with no validation**

The `total_chars // 4` estimate in `Agent._loop()` (lines 90–93) has no baseline validation against any model's actual tokeniser. The `context_window` config key (line 95) has no default per-model values and is not surfaced in `config.toml` or the UI.

**Status:** Open. The warning fires but no enforcement or user-facing feedback exists.

### Open questions

- **Should `context_window` be a required config key per model, or inferred from Ollama's model info API?** The current warning is silent and advisory. If Ollama exposes context window size via its `/api/show` endpoint, it could be fetched at startup and stored in config, removing the need for manual configuration.

- **What is the actual token estimation error margin for `qwen2.5:1.5b` at the character-count heuristic?** The `total_chars // 4` figure is used without calibration. Running a sample of real sessions through Ollama's tokeniser would establish whether 80% is a safe threshold or too conservative/aggressive.

- **Should the `history_limit` key be exposed in the UI?** It is configurable via `config.toml` but there is no setting screen for it. Users have no discoverable way to change it.

- **Is `call_after_refresh` a reliable deferral mechanism across all Textual versions in use?** The fix for `_test_on_mount` depends on `call_after_refresh` firing after `MainScreen`'s full compose cycle. Textual's rendering pipeline could, in principle, interleave refreshes and mounts in ways that still race under load. The test passes consistently in this environment but has not been tested under slower machines or different Textual versions.

- **What did the previous agent session change in `stash/tui/widgets/sidebar.py` beyond the ID mismatch?** The last commit reports `12 ++++++----` lines changed in that file, but the only outstanding diff against HEAD is the one-line ID normalisation fix. The other changes are already committed and were not reviewed in isolation this session.

### Decisions taken

**`ctrl+k` as the new locations binding**

Alternatives considered: disabling Textual's command palette (`ENABLE_COMMAND_PALETTE = False`) to reclaim `ctrl+p`, or rebinding to another key. Disabling the palette was deferred as a larger change with unclear side-effects. `ctrl+k` was chosen as it is unused across the entire codebase and not a standard terminal shortcut within a Textual TUI context.

**Location registry emphasis kept to one place in `test_notes.md`**

An initial draft duplicated the explanation of `ctrl+k` / location registry across three sections (standalone section, keybinding callout, and test cases). Collapsed to a single sentence under the keybindings table and kept the test cases, removing the standalone section.

### Problems identified

**`ctrl+p` for locations never appeared in the footer**

`StashApp` does not set `ENABLE_COMMAND_PALETTE = False`, so Textual's built-in `ctrl+p` → command palette binding shadowed the custom `open_locations` binding. The footer displayed the palette entry and the locations entry was invisible. The binding was never functional as `ctrl+p` in practice.

**Status:** Resolved. Binding changed to `ctrl+k`.

**Ollama running at 4,096 context instead of 32,768**

Ollama's startup logs showed `n_ctx_seq (4096) < n_ctx_train (32768)`. The model is trained on 32,768 tokens but Ollama allocates only a 4,096-token KV cache by default on memory-constrained machines. The `context_window` config was defaulting to 32,768, making the 80% pressure warning fire at ~26,000 estimated chars — well above the real 4,096-token limit. The warning would never fire even as the model ran out of context.

**Status:** Open for the user's machine (hardware-constrained, cannot allocate more). Documented in `test_notes.md` with instructions for the tester to match `context_window` to their actual `n_ctx_seq`.
